# ERI DB Search — локальная база составных артикулов (ER + 4 цифры)

Web-приложение для импорта `.xlsx`, разделения строк на **базовые артикулы** (`ER…`) и **добавочные коды** (`0000`…`9999`), материализации всех **составных** комбинаций `ER…-0001` и быстрого поиска по ним.

## Стек

- **Frontend:** React 18, TypeScript, Vite  
- **Backend:** Node.js, Express, TypeScript  
- **БД:** SQLite ([better-sqlite3](https://github.com/WiseLibs/better-sqlite3))  
- **Excel:** [ExcelJS](https://github.com/exceljs/exceljs)

## Модель данных

| Таблица | Назначение |
|--------|------------|
| `base_articles` | Базовые строки: 2-й столбец начинается с `ER`, имя из 4-го столбца |
| `add_articles` | Добавочные строки: 2-й столбец — ровно 4 цифры, имя из 4-го (с очисткой ведущих запятых) |
| `search_variants` | Материализованные пары база×добавка: `composite_art_original`, `composite_art_normalized`, `display_name`, … |
| `search_variants_fts` | FTS5 по нормализованному составному артикулу и именам |

Для **каждого файла** после импорта: все базы этого файла декартово умножаются на **уникальный набор добавок** того же файла (при дублях кода `0001` берётся строка с большим `id`). Повторный импорт того же файла не создаёт дубликатов строк (уникальность по `filename + sheet + row`) и не плодит новые варианты для уже существующих пар.

## Импорт (.xlsx)

1. Определяется строка заголовка (как раньше — по плотности заполненных ячеек в первых 15 строках).  
2. **Столбец 2** (колонка B) — артикул; **столбец 4** (D) — наименование.  
3. Строка классифицируется:
   - начинается с `ER` → база;
   - ровно 4 цифры → добавка;
   - иначе — в лог ошибок, строка пропускается.  
4. Пустой 2-й столбец → ошибка на строку, пропуск.  
5. После вставки баз и добавок для файла строятся все `search_variants`.

## Поиск

- Запрос нормализуется: `trim`, верхний регистр, удаление пробелов и прочих не-буквоцифровых символов → один ключ, совместимый с вводом **с дефисом**, **без дефиса** и с **пробелами** (`ER010000000001-0001`, `ER0100000000010001`, `ER010000000001 - 0001`).  
- Приоритет: точное совпадение по `composite_art_normalized` → совпадение «компактной» формы оригинала → префикс → подстрочный поиск по именам → FTS.

## Быстрый старт

```bash
npm install
npm run dev
```

Браузер: **http://127.0.0.1:5173** (прокси `/api` → **8787**).

### Пример файла

```bash
npm run sample:xlsx
```

Файл: `data/sample-import.xlsx` — загрузите через UI.

### Проверка составного артикула

После импорта примера выполните в UI или через API:

- `ER010000000001-0001`  
- `er010000000001-0001`  
- `ER0100000000010001`  
- `ER010000000001 - 0001`  

Ожидается одна запись с итоговым именем: базовое наименование + `, ` + очищенное имя добавки (`КВБ12, КВБ12`).

## Переменные окружения

См. `.env.example`: `PORT`, `DATABASE_PATH`, `CLIENT_DIST`.

## Миграция со старой схемы

При первом запуске с новой версией таблицы `items` / `items_fts` удаляются автоматически (см. `server/src/db.ts`). Старый файл SQLite можно также удалить вручную: `data/articles.sqlite`.

## Ограничения

- Файл в памяти (лимит multer ~40 МБ).  
- Добавки с одинаковым кодом в одном файле: для вариантов используется одна «каноническая» строка добавки (последняя по `id` среди дублей кода).
